# 🔄 업무 부서 전달(이관) 기능 도입 논의

## 1. 개요
현재 프로젝트에는 AI가 손님의 요청을 분석하여 부서(하우스키핑, 시설 등)에 자동 배정하는 기능이 있습니다. 
하지만 **AI에 의해 잘못 배정된 업무를 올바른 부서로 재배정(이관)하는 기능이 아직 구현되어 있지 않습니다.** 
이에 대해 기능을 추가할지, 우선순위를 어떻게 가져갈지 논의하기 위한 문서입니다.

---

## 2. 기능의 필요성 (왜 필요한가?)

### 1) AI 분류 한계에 대한 안전망 (Fallback)
- AI 분류가 100% 완벽할 수 없기 때문에 잘못 배정되는 상황이 반드시 발생합니다. (예: "방이 추워요" → 하우스키핑으로 배정되지만 실제로는 시설팀 수리 필요)
- 버튼 한 번으로 해당 부서에 업무를 토스하여 문제를 신속하게 해결할 수 있습니다.

### 2) 데이터 이력 단절 방지
- 이관 기능이 없으면, 잘못 배정된 기존 업무를 **'취소'**하고 **'새로 등록'**해야 합니다.
- 이 경우 손님이 처음에 요청했던 시간 정보가 초기화되어, 손님이 얼마나 오래 기다렸는지 정확한 지연 시간(Overdue) 추적이 불가능해집니다.

### 3) 책임 소재 명확화 및 누락 방지
- 시스템이 아닌 무전기/구두로 "이건 시설팀에서 해주세요"라고 요청하면 업무가 누락되기 쉽습니다.
- 시스템 상에서 이관 시, 대상 부서의 '대기열(Pending)'로 명확히 넘어가므로 책임 소재가 분명해집니다.

### 4) AI 모델 재학습 데이터 확보
- 관리자가 수동으로 부서를 이관한 내역은 "AI의 오답 노트"가 됩니다.
- 이를 바탕으로 향후 AI 모델 고도화 시 매우 유용한 피드백 데이터로 활용할 수 있습니다.

---

## 3. 기능 미구현 시 대안 (Workaround)
MVP(최소 기능 제품) 시연 시나리오상 당장 개발할 시간이 부족하다면, 아래와 같이 우회할 수는 있습니다.
- **처리 방법:** 잘못 배정받은 부서의 직원이 해당 요청을 `취소` 처리하고, 관리자나 직원이 올바른 부서를 지정해 `새 요청을 수동 등록`합니다.
- **단점:** 절차가 번거로우며, 사용자(직원) 경험이 크게 저하되고 대기 시간 통계가 왜곡됩니다.

---

## 4. 기술적 구현 방안 요약
이미 핵심 도메인 로직은 백엔드에 준비되어 있어 구현 난이도가 높지 않습니다.

* **백엔드 (Backend)**
  * `Request` 엔티티 내에 이미 존재하는 `transferDepartment()` 메서드 활용.
  * `ChangeDepartmentCommand` DTO 생성 및 `ManageAdminRequestUseCase` 확장.
  * `AdminRequestController`에 부서 변경 API (`PATCH /api/admin/requests/{id}/department`) 추가.
* **프론트엔드 (Frontend)**
  * BFF 패턴에 맞춰 `app/api/admin/requests/[id]/department/route.ts` API 라우트 추가.
  * `useChangeDepartment` 훅 작성.
  * 관리자 상세 뷰 또는 목록 뷰에 [부서 재배정] 버튼 및 선택 모달 UI 추가.

## 5. 팀원 논의 필요 사항
내일 회의 시 아래 안건에 대해 결정하면 좋습니다.

1. **우선순위 결정:** 
   - 지금 바로 이 기능 개발에 착수할 것인지?
   - 아니면 다른 중요한 결함/기능(예: 채팅 안정화, 기본 로직 완성 등)을 먼저 끝낼 것인지?
2. **UI/UX 정책:**
   - 부서를 이관할 때 "이관 사유"를 간단히 메모로 남기게 할 것인지? (선택 사항)
   - 이관 권한을 '관리자(Admin)'에게만 줄 것인지, '일반 직원(Staff)'도 본인 부서로 잘못 온 것을 이관할 수 있게 할 것인지?

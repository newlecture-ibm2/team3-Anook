## 🔍 지연 원인 (Bottleneck) 분석

현재 파이프라인은 한 번의 채팅 요청에 대해 **다수의 LLM API를 직렬(Sequential)로 호출**하는 구조적 병목을 가지고 있습니다.  
다중 요청(예: "수건 2개 주시고요 에어컨 안 나와요") 발생 시 연쇄적인 대기 시간이 발생합니다.

### 1. RAG 임베딩 무조건 검색 병목 (약 0.3 ~ 0.5초)
- 메시지가 들어오면 의도와 상관없이 무조건 RAG(정보 검색)용 임베딩 API를 호출하고 벡터 DB를 검색합니다.
- 단순 업무 지시(예: "수건 줘")에서도 불필요한 대기 시간이 발생합니다.

### 2. 라우터 엔진 호출 (약 1.5 ~ 2.0초)
- RAG 매칭 실패 후, 의도 파악을 위해 라우터 모델(`gemini-2.5-flash`)을 호출합니다.

### 3. 부서별 에이전트 직렬 호출 병목 (약 4.0 ~ 5.0초) 🚨 **가장 큰 병목**
- 라우터가 여러 부서(예: `[HK, FACILITY]`)를 반환하면, `analyze.py`의 `for` 루프에서 **순차적으로 대기하며** 에이전트를 호출합니다.
- HK 에이전트 완료(2초 대기) 후 FACILITY 에이전트를 호출(2초 대기)하므로, API 네트워크 지연이 그대로 누적됩니다.

> **결과**: 임베딩 + 라우터 + HK 에이전트 + FACILITY 에이전트 등 여러 API 호출이 누적되어 평균 **6~8초** 이상의 응답 지연이 발생합니다.

### 현재 흐름도

```
[Guest 메시지]
    │
    ▼
STEP 0. 긴급 키워드 Pre-Filter ──── 히트 → 즉시 응답 (0초)
    │
    ▼
STEP 1. COMMON RAG 검색 (0.3~0.5초) ── 히트 → 즉시 응답
    │ 미스
    ▼
STEP 2. 라우터 엔진 (1.5~2초) ──────── [HK, FACILITY] 반환
    │
    ▼
STEP 3. 에이전트 순차 호출 🚨
    ├── HK Agent  (2초 대기)
    └── FACILITY Agent (2초 대기)  ← 총 4초 (직렬)
    │
    ▼
[응답 반환] ─── 총 ~6~8초
```

## ✅ 적용 완료 — 에이전트 비동기 병렬 처리 (Asyncio Gather)

| 항목 | 내용 |
|------|------|
| **상태** | 🟢 구현 완료 |
| **브랜치** | `young/feat/AN-279/Agent-Async` |
| **기대 효과** | 다중 인텐트 시 ~2초 단축 (6초 → 4초) |
| **변경 범위** | AI 서버만 (백엔드/프론트 변경 없음) |
| **리스크** | 중 — ContextVar 충돌 해결 필요 |

### 개념

라우터가 반환한 여러 도메인 에이전트를 `for`문으로 기다리지 않고, Python의 `asyncio.gather()`를 활용해 **동시에(병렬로) 호출**합니다.

### 변경 대상 파일

| 파일 | 변경 내용 |
|------|-----------|
| `ai/app/infrastructure/gemini/client.py` | `call_gemini_async()` 비동기 래퍼 함수 추가 |
| `ai/app/core/hk_engine.py` | `async def` 전환 + `await call_gemini_async()` |
| `ai/app/core/facility_engine.py` | 동일 패턴 |
| `ai/app/core/concierge_engine.py` | 동일 패턴 |
| `ai/app/core/fb_engine.py` | 동일 패턴 |
| `ai/app/core/emergency_engine.py` | 동일 패턴 |
| `ai/app/api/analyze.py` | STEP 3 for 루프 → `asyncio.gather()` 병렬화 |

### 개선 후 흐름도

```
STEP 3. 에이전트 병렬 호출 ✅
    ├── HK Agent  ──────┐
    └── FACILITY Agent ──┤  (동시 실행: max 2초)
                         ▼
              [응답 반환] ─── 총 ~4초
```

### ⚠️ 주의: ContextVar 충돌 문제

`call_gemini()` 내부에서 `ai_log_meta_ctx.set()`으로 로깅 메타데이터를 전역 저장하고 있습니다.  
병렬 실행 시 **두 에이전트의 로그가 서로 덮어쓰여** AI 로그가 뒤섞이거나 유실될 수 있습니다.

**해결 방안**: 각 에이전트 호출 시 `copy_context().run()`으로 독립 컨텍스트를 만들거나, 메타데이터를 함수 반환값에 포함시키는 방식으로 변경해야 합니다.

## ✅ 적용 완료 — 중간 진행 상태 피드백 (Progress Indicator)

| 항목 | 내용 |
|------|------|
| **상태** | 🟢 구현 완료 |
| **브랜치** | `young/feat/AN-279/Agent-Async` (동일 브랜치) |
| **기대 효과** | 체감 대기 시간 ~3초 단축 |
| **변경 범위** | AI 서버 + 백엔드 + 프론트엔드 (3계층) |
| **리스크** | 낮음 — 기존 로직에 영향 없는 순수 추가 기능 |

### 개념

백엔드 성능 개선 전, 고객이 느끼는 '체감 대기 시간'을 줄이는 UX 기법입니다.  
라우터 처리가 끝나는 시점(약 1.5초)에 WebSocket으로 중간 상태 메시지를 프론트엔드에 먼저 전송합니다.

### 고객 화면 예시

```
┌─────────────────────────────────┐
│  🔄 AI가 하우스키핑, 시설관리    │

│                    ← (pulse 애니메이션)
└─────────────────────────────────┘
      ↓ (2초 후 자동 교체)
┌─────────────────────────────────┐
│  ✅ 수건 2개 요청이 접수되었습니다│
│  ✅ 에어컨 수리 요청을 전달합니다 │
└─────────────────────────────────┘
```

### 데이터 흐름

```

    │  POST /chat/{roomNo}/progress  { domains: ["HK","FACILITY"] }

    │  type: "AI_RESPONSE" 수신 → ProgressIndicator 자동 제거
```

## 🔮 추후 개선 가능 사항 (팀 협의 필요)

아래 항목들은 추가적인 성능 개선이 가능하지만, **팀원의 코드에 영향을 주거나 트레이드오프가 있어** 협의 후 진행해야 합니다.

---

### A. STEP 1(COMMON RAG) ↔ STEP 2(라우터) 병렬 실행

| 항목 | 내용 |
|------|------|
| 기대 효과 | ~0.5초 절약 |
| 난이도 | ⭐ 낮음 |
| 리스크 | 🟡 중간 |

**개념**: 현재 RAG 검색(STEP 1)과 라우터(STEP 2)는 순차 실행이지만, 서로 의존성이 없으므로 `asyncio.gather()`로 동시 실행 가능합니다.  
RAG가 히트하면 라우터 결과를 버리고 즉시 응답합니다.

**⚠️ 잠재 문제**:
- COMMON RAG 히트율이 높으면(30% 이상) 라우터 API 호출이 **비용 낭비**가 됩니다.
- 히트율이 10% 미만이면 병렬화 이득 > 낭비이므로 적용 가치가 있습니다.

**📋 팀 판단 포인트**: 현재 COMMON RAG의 히트율이 어느 정도인지 로그로 측정한 후 결정.

---

### B. Lazy RAG (부서별 RAG 지연 호출)

| 항목 | 내용 |
|------|------|
| 기대 효과 | TASK 요청 시 ~0.3초 절약 |
| 난이도 | ⭐ 낮음 |
| 리스크 | 🔴 높음 (역효과 가능) |

**개념**: 라우터가 먼저 `mode: INFO`라고 판별했을 때만 RAG 검색을 실행하여, "수건 줘" 같은 TASK 요청에서 불필요한 임베딩 호출을 차단합니다.

**⚠️ 잠재 문제**:
- **INFO 질문의 응답이 오히려 느려짐**: 현재는 COMMON RAG 히트 시 라우터를 건너뛰고 0.5초 만에 즉시 응답하지만, Lazy RAG로 바꾸면 라우터(1.5초) → RAG(0.5초) = 2초로 증가합니다.
- "조식 몇 시에요?" 같은 자주 묻는 질문이 1.5초 더 느려지는 체감이 발생합니다.

**📋 팀 판단 포인트**: COMMON RAG는 기존대로 유지하고, 에이전트 내부의 부서별 RAG만 Lazy로 전환하는 **절충안**을 검토.

---

### C. FB 에이전트 내부 병렬화 (메뉴 API ↔ RAG)

| 항목 | 내용 |
|------|------|
| 기대 효과 | FB 에이전트 ~0.3초 절약 |
| 난이도 | ⭐ 낮음 |
| 리스크 | 🟢 낮음 (가장 안전) |

**개념**: `fb_engine.py`에서 `_fetch_menu_context()`(백엔드 메뉴 API 호출)와 `rag_service.search_similar()`(RAG 검색)를 `asyncio.gather()`로 동시 실행합니다.

```python
# 현재: 순차 실행 (0.3초 + 0.3초 = 0.6초)
menu_context = _fetch_menu_context()
rag_results = rag_service.search_similar(...)

# 개선: 병렬 실행 (max(0.3초, 0.3초) = 0.3초)
menu_context, rag_results = await asyncio.gather(
    asyncio.to_thread(_fetch_menu_context),
    asyncio.to_thread(rag_service.search_similar, ...)
)
```

**⚠️ 잠재 문제**: 거의 없음. 두 호출은 완전히 독립적이며, 기존 예외 처리도 유지됩니다.

**📋 팀 판단 포인트**: FB 에이전트 담당 팀원이 작업 1(에이전트 async 전환) 이후에 바로 적용 가능.

### D. 라우터 경량 모델 적용 (gemini-2.0-flash-lite)

| 항목 | 내용 |
|------|------|
| 기대 효과 | 라우터 단계 ~0.5초 절약 |
| 난이도 | ⭐ 낮음 (코드 1줄) |
| 리스크 | 🟡 중간 (정확도 검증 필수) |

**개념**: 라우터는 "분류 작업"이므로 현재의 `gemini-2.5-flash`보다 경량 모델로도 충분할 수 있습니다.

```python
# router_engine.py — 변경 1줄
model_name="gemini-2.0-flash-lite"  # 현재: gemini-2.5-flash
```

**⚠️ 잠재 문제**:
- **다중 인텐트 파싱 정확도 하락**: "수건 2개 주시고 에어컨도 봐주세요"를 `[HK, FACILITY]` 2개로 분리하는 능력이 떨어질 수 있습니다.
- JSON 형식이 깨지거나, `confidence` 값이 부정확해져 불필요한 FRONT Fallback이 발동할 수 있습니다.

**📋 팀 판단 포인트**: 모델 교체 전 **기존 테스트 케이스로 A/B 정확도 비교**를 반드시 수행한 후 결정.

### E. 프롬프트 토큰 최적화

| 항목 | 내용 |
|------|------|
| 기대 효과 | 에이전트당 ~0.2초 절약 |
| 난이도 | ⭐⭐ 중간 (팀 조율 필요) |
| 리스크 | 🟡 중간 |

**개념**: 각 부서 에이전트의 시스템 프롬프트에서 과도한 few-shot 예시를 줄여 입력 토큰 수를 줄이면, Gemini 응답 시간이 줄어듭니다.

**⚠️ 잠재 문제**:
- 각 부서 프롬프트(`hk_prompt.py`, `fb_prompt.py` 등)는 **해당 도메인 담당 팀원이 관리**하고 있습니다.
- 프롬프트를 임의로 수정하면 에이전트 정확도가 떨어지거나, Git 충돌이 발생할 수 있습니다.

**📋 팀 판단 포인트**: 각 부서 담당자가 자기 프롬프트를 검토하고, 불필요한 반복 예시만 정리하는 방식으로 진행.

### F. 모델 통합 — 라우터 + 에이전트 원샷 처리

| 항목 | 내용 |
|------|------|
| 기대 효과 | ~4초 절약 (가장 큰 효과) |
| 난이도 | ⭐⭐⭐ 높음 |
| 리스크 | 🔴 높음 (아키텍처 변경) |

**개념**: 프롬프트 엔지니어링을 극대화하여 "단 1번의 LLM 호출"로 도메인 라우팅과 엔티티 추출을 모두 끝냅니다.

**⚠️ 잠재 문제**:
- 프롬프트가 비대해져 모델 인지력이 떨어질 수 있습니다.
- **팀원들이 부서별 플러그인을 독립적으로 개발하는 마이크로 에이전트 아키텍처의 장점을 포기**해야 합니다.
- 현재 팀 운영 방식과 상충되므로 전체 아키텍처 재설계가 필요합니다.

**📋 팀 판단 포인트**: 현재 팀 구조에서는 비추천. 향후 프로젝트 규모가 축소되거나 유지보수 단계에 들어갔을 때 재검토.

# 🏨 Anook AI 라우팅 및 핑퐁(Human-in-the-Loop) UX 가이드

본 문서는 Anook 호텔 컨시어지 AI의 **[메인 라우터 ➡️ 부서 에이전트 ➡️ 백엔드 ➡️ 프론트엔드]**로 이어지는 전체 아키텍처와, 모호한 고객 요청을 처리하는 핑퐁(되묻기) UX의 설계 기준을 명확히 정의합니다.

---

## 1. AI 프롬프트 아키텍처

우리 AI 시스템은 **메인 라우터**와 **부서별 에이전트**로 완벽하게 역할이 분리되어 있습니다.

### 🏢 메인 라우터
* **담당 파일:** `router_prompt.py`
* **역할:** 고객의 메시지를 가장 먼저 수신하고 분석하는 1차 관문.
* **하는 일:** 대화의 목적을 5가지(TASK, CHITCHAT, CLARIFICATION, INFO, CANCEL) 중 하나로 분류하고, TASK인 경우 6개 부서(HK, FB, FACILITY, CONCIERGE, FRONT, EMERGENCY) 중 하나로 라우팅을 수행합니다.

### 🤖 부서별 에이전트
* **담당 파일:** `hk_engine.py`, `front_engine.py` 등 (`DOMAIN_AGENTS`에 등록된 에이전트들)
* **역할:** 라우터가 배정한 특정 부서의 고객 요청만 전담하여 세부 처리를 담당.
* **하는 일:** 자신의 부서에 맞는 세부 정보(`entities`)를 깊게 파고들어서 추출합니다. (예: 메뉴, 수량, 연장 시간 등)

---

## 2. 핑퐁 (Clarification) 옵션 발동 기준과 로직

고객의 요청이 모호할 때 AI가 지레짐작하지 않고 선택지 버튼(Pill)을 띄우는 **핑퐁 UX**는 2가지 상황에서 발동합니다.

### A. 라우터 레벨 핑퐁 (부서 자체가 헷갈릴 때)
* **상황:** 고객이 "물 좀 줘" 라고 했는데, 이게 하우스키핑(생수병 무료 제공)인지 식음료(룸서비스 얼음물)인지 라우터가 판단할 수 없을 때.
* **로직:** 
  1. 라우터가 자체적으로 `mode: CLARIFICATION`을 발동.
  2. 라우터가 질문(`clarification_message`)과 선택지(`clarification_options`: `["생수병", "얼음물"]`)를 생성.
  3. `domain_code`가 `null`로 처리되어 **직원 대시보드에는 업무 카드가 생성되지 않음.** (직원 피로도 제로)

### B. 부서 레벨 핑퐁 (부서는 맞는데 디테일이 부족할 때)
* **상황:** "체크아웃 늦출게요" (프론트 업무인 건 확실하나, '몇 시'인지 빠져 있을 때)
* **로직:**
  1. 라우터는 정상적으로 `FRONT` 부서로 안내(`mode: TASK`).
  2. 프론트 에이전트가 자체적으로 질문(`guest_reply`)과 선택지(`clarification_options`: `["12시", "1시"]`)를 생성.
  3. 부서가 확정되었으므로 **직원 대시보드에 PENDING 상태의 카드가 생성**됨.

> 💡 **데이터 흐름 (공통):** 라우터든 부서든 `clarification_options` 배열에 데이터를 담으면, 중앙 통제소(`analyze.py`)가 이를 추출하여 자바 백엔드로 넘기고, 웹소켓을 거쳐 리액트 프론트엔드의 `<ChatScreen />`에서 `<Pill />` 버튼으로 짠! 렌더링됩니다.

## 3. 부서별 "해결 불가 (Confidence 0.4 미만)" 기준과 글로벌 Fallback

각 부서 에이전트는 무적의 만물상자가 아닙니다. 부서마다 "내 소관이 아니다(해결 불가)"라고 판단하는 기준이 다르며, 이 기준은 **각 부서의 프롬프트 내에 명확한 지시문으로 정의**됩니다.

### 📌 프롬프트 작성 예시 (해결 불가 기준)
* **HK(하우스키핑) 프롬프트:** *"만약 고객이 수건, 청소, 어메니티가 아닌 요금 결제나 룸서비스를 요구한다면, 넌 처리할 수 없으므로 `confidence`를 0.2로 설정해라."*
* **FRONT(프론트) 프롬프트:** *"만약 고객이 체크아웃이나 요금 문의가 아니라, 방에 벌레가 나왔다고 하거나 변기가 막혔다고 하면, 네 소관이 아니므로 `confidence`를 0.3으로 설정해라."*

### 🤔 왜 부서가 직접 다른 부서로 라우팅하지 않고 점수를 낮추어 포기하나요?
만약 부서 에이전트가 "이건 내 일이 아니니까 식음료(FB)로 보내야지" 하고 직접 타 부서 라우팅을 시도하게 두면 다음과 같은 **치명적인 설계 결함**이 발생합니다.
1. **무한 핑퐁 지옥 (Infinite Routing Loop):** 메인 라우터가 HK로 보냄 ➡️ HK가 FB로 넘김 ➡️ FB가 다시 HK로 넘김 ➡️ AI들끼리 무한 반복.
2. **단일 책임 원칙(SRP) 위배:** 하우스키핑 에이전트가 식음료, 시설, 프론트 등 다른 모든 부서의 역할을 전부 학습하고 있어야 합니다.

**✅ 해결책 (우아한 항복):** 
부서 에이전트는 오직 "내가 할 수 있다(처리)" 또는 "내가 할 수 없다(항복: `confidence < 0.4`)" 두 가지만 판단합니다. 부서 에이전트가 항복하면, AI들끼리 싸우게 두지 않고 **즉시 인간 직원(프론트데스크)에게 안전하게 에스컬레이션**하여 라우팅 충돌과 무한 루프를 원천 차단합니다.

### 🚨 중앙 통제소의 글로벌 Fallback (강제 이관 로직)
부서 에이전트가 위 프롬프트 지시를 읽고 "아, 내 일 아니네" 하며 `confidence < 0.4`를 뱉어내면, 각 부서는 직접 이관 코드를 짤 필요가 없습니다. 

**중앙 통제소(`analyze.py`)가 모든 에이전트의 결과를 마지막에 검사하여 알아서 프론트로 던져줍니다!**

```python
# analyze.py 내 글로벌 이관 핵심 로직
if final_confidence < 0.4:
    final_domain_code = "FRONT"            # 원래 부서를 무시하고 무조건 FRONT로 이관
    final_entities["intent"] = "ESCALATION"  # 의도를 '에스컬레이션'으로 강제 변경
    final_guest_reply = "죄송합니다. 정확한 파악이 어려워 즉시 프런트 데스크 직원에게 연결해 드리겠습니다."
```

### ✅ 요약: 부서 에이전트 개발자의 행동 수칙
앞으로 새로운 부서(예: CONCIERGE) 에이전트를 추가할 팀원은 파이썬 이관 코드를 건드릴 필요가 없습니다. 그저 **해당 부서의 프롬프트 파일 맨 밑에 "네 소관이 아니면 confidence를 낮게 줘라" 한 줄만 적어두면 전체 시스템이 알아서 작동**합니다.

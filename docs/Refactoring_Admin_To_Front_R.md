# 📌 [Architecture] Admin 역할을 Front Desk로 리네이밍하는 이유

현재 우리 시스템의 `Admin(관리자)` 역할을 `Front Desk(프론트 데스크)`로 전면 개편(리네이밍)하고자 합니다. 이번 리팩토링의 목적은 **도메인 주도 설계(DDD) 원칙을 준수하여, 실제 호텔 비즈니스 로직과 시스템 아키텍처를 완벽하게 일치시키는 것**입니다.

---

## 1. 비즈니스 도메인과 시스템 용어의 일치 (Ubiquitous Language)
소프트웨어 시스템에서 'Admin(관리자)'이라는 용어는 보통 계정 생성, 권한 부여, 시스템 설정 등을 담당하는 **IT 시스템 관리자**를 의미합니다. 

하지만 현재 우리 아늑(Aneuk) 서비스에서 'Admin'이 수행하는 역할은 실제 호텔의 **프론트 데스크 또는 당직 지배인(Duty Manager)**의 역할입니다.
* 하우스키핑, F&B 등 타 부서의 작업 지연 모니터링
* 예외 상황 발생 시 에스컬레이션 승인 및 수동 배정 처리
* 긴급 대응 상황 컨트롤 타워 역할

따라서 시스템의 역할을 실제 비즈니스 용어인 `Front Desk`로 변경함으로써, 코드를 읽는 누구나 해당 역할의 책임과 권한을 직관적으로 이해할 수 있게 됩니다.

## 2. 부서(Department)와 권한(Privilege)의 명확한 분리
우리 시스템의 부서 목록(하우스키핑, F&B, 시설 등)에 '프론트 데스크'가 포함되어 있는 것은 매우 정상적이고 올바른 구조입니다. 프론트 데스크 역시 '레이트 체크아웃', '객실 키 분실' 등 고유의 업무를 할당받고 처리하는 하나의 부서이기 때문입니다.

일반 부서와 프론트 데스크의 유일한 차이는 **'시스템 통제 권한'**에 있습니다.
* **일반 부서 (Staff)**: 자신이 속한 부서의 요청만 확인하고 처리
* **프론트 데스크 (Front Desk)**: 자신의 부서 요청을 처리함과 동시에, **호텔 전체 시스템을 조망하고 타 부서를 통제할 수 있는 권한** 보유

이를 아키텍처로 깔끔하게 분리하기 위해 DB 설계를 아래와 같이 변경합니다.
> `department` 테이블의 기존 `is_admin` 컬럼을 `is_front_desk`로 변경.
> 부서 데이터 중 '프론트데스크' 부서만 `is_front_desk = TRUE`를 가지며, 이 부서에 속한 직원만이 `ROLE_FRONT_DESK` 권한을 부여받아 전체 대시보드에 접근 가능.

## 3. 향후 확장성 확보
현재 '호텔 운영의 관리자(Front Desk)'를 '시스템 관리자(Admin)'라는 이름으로 섞어 쓰게 되면, 추후 진짜 IT 시스템 전반(라이선스 관리, 테넌트 관리, 시스템 환경 변수 등)을 관리해야 하는 **'슈퍼 관리자(Super Admin)'** 도입 시 권한 충돌과 의미적 혼란이 발생합니다.

미리 운영 주체를 `Front Desk`로 명확히 분리해 두면, 향후 시스템 전용 관리자가 필요해질 때 아키텍처의 변경 없이 권한만 추가하면 되는 확장성을 얻게 됩니다.

---

## 🚀 리팩토링 주요 변경 범위
본 리팩토링은 로직의 변경이 아닌 **네이밍과 경로의 시맨틱(의미) 개선**에 집중합니다.

1. **DB 및 보안 설정**: `is_admin` ➡️ `is_front_desk`, 권한 `ROLE_ADMIN` ➡️ `ROLE_FRONT_DESK`
2. **API 및 프론트 라우트**: `/admin/...` ➡️ `/front-desk/...`
3. **Java 패키지 및 클래스명**: `com.anook.backend.admin.*` ➡️ `com.anook.backend.frontdesk.*`, `Admin*Controller` ➡️ `FrontDesk*Controller`
4. **WebSocket 채널**: `/topic/admin` ➡️ `/topic/front-desk`

이 작업을 통해 우리의 코드베이스는 실제 호텔 비즈니스 도메인을 더 정확하게 반영하는 탄탄한 아키텍처로 거듭날 것입니다.
